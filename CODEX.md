# NodeClassification 项目说明

本文件是给 Codex 阅读的项目说明文件，用于约束后续实现、修改和维护本仓库时的默认判断。仓库当前目标是完成研究生课程《机器学习》的期末大作业，主题为图上的节点分类任务。除非用户另有说明，后续实现应围绕 `dataset/` 中已有数据集展开，优先保持代码结构清晰、实验流程可复现、说明文字适合课程报告或论文式表述。

## 项目目标与任务背景

本项目研究的问题是图结构数据上的节点分类。与普通表格数据分类不同，图节点分类中的每一个样本不仅具有自身属性，还处在由边连接形成的拓扑结构中；模型需要综合利用节点特征、邻居节点信息以及图的整体连接模式，预测每个节点所属类别。在当前数据集中，节点通常表示论文，边表示论文之间的引用关系，标签表示论文主题类别。因此，该任务可以被理解为在学术引用网络中，根据论文的词袋特征和引用网络结构推断论文研究方向。

从机器学习问题建模角度看，节点分类具有典型的关系学习特征。一个节点的类别并不是完全由其局部特征独立决定，相邻节点或高阶邻居往往携带与其类别相关的上下文信息。后续模型设计应体现这一点：即使实现基线方法，也应清楚区分仅使用节点特征的普通分类器和利用图结构的图神经网络方法，例如 GCN、GraphSAGE、GAT 或其他消息传递模型。实验报告中也应说明图结构信息对分类性能的作用，而不是只把数据当作普通矩阵处理。

## 任务难度说明

本项目不能被写成类似“鸢尾花分类”的简单入门任务。鸢尾花数据集通常只有 150 个样本、4 个连续特征和 3 个类别，样本之间被默认视为相互独立，经典线性模型、决策树或支持向量机即可快速得到较高准确率。该类任务主要用于演示监督学习基本流程，其难点集中在特征输入、模型调用和指标计算，无法充分体现复杂数据结构建模、半监督学习或深度模型训练过程。

相比之下，图节点分类要求模型处理非独立同分布样本。节点之间的边会导致训练样本、验证样本和测试样本在结构上相互关联，模型既要利用这种关联，又要避免不合理的数据泄漏。图模型通常涉及邻接矩阵或边索引构造、特征归一化、消息传递、邻居聚合、过平滑风险、训练/验证/测试划分以及随机种子控制等问题。尤其在 Cora、Citeseer 和 Pubmed 等引用网络上，节点数量、边数量和特征维度均显著高于鸢尾花任务，实验过程也需要比较不同模型、不同超参数和不同数据集上的泛化表现。因此，本项目在数据结构、算法设计、工程实现和实验分析层面均明显高于简单入门分类项目的要求。

## 数据集与当前加载约定

数据集统一放在 `dataset/` 目录下，当前包含 `cora`、`citeseer` 和 `pubmed` 三个引用网络数据集。每个数据集目录包含原始文本数据、预处理脚本和预处理后的中间文件。原始数据通常位于 `origin/*.content` 与 `origin/*.cites` 中，其中 `.content` 文件保存节点 ID、词袋特征和类别标签，`.cites` 文件保存引用边。`remake.py` 会从原始文件生成 `features.npy`、`id_map.json` 和 `class_map.json`，后续训练代码应优先复用这些已生成文件。

当前仓库已有的 `GraphData.LoadfromSet()` 逻辑会使用 `id_map.json` 建立原始节点 ID 到连续整数 ID 的映射，读取 `<dataset>.cites` 构造 NetworkX 无向图，读取 `features.npy` 得到节点特征并做按行归一化，读取 `class_map.json` 得到节点标签和类别到节点集合的映射。后续如果重构数据模块，应保留这些语义，并明确处理节点编号、特征矩阵、边索引、标签向量和数据划分之间的一致性。

当前数据规模应按仓库内文件为准：Cora 包含 2708 个节点、5429 条引用边和 1433 维特征；Citeseer 包含 3312 个节点、4715 条引用边和 3703 维特征；Pubmed 包含 19717 个节点、44338 条引用边和 500 维特征。实现实验时，应至少支持在这些数据集之间通过参数切换，并保证不同数据集的类别数、特征维度和图规模不被硬编码。

## 后续代码结构要求

当前仓库尚未形成完整训练工程。后续实现必须保持功能模块清晰，至少拆分为参数、模型、训练器、工具函数和主入口等职责。可以采用单文件形式，如 `args.py`、`model.py`、`trainer.py`、`utils.py`、`main.py`，也可以采用同名包目录，如 `args/`、`model/`、`trainer/`、`utils/` 和 `main.py`；一旦选择一种结构，应在整个项目中保持一致。

`args` 模块负责集中定义命令行参数和默认超参数，包括数据集名称、数据目录、模型类型、隐藏维度、学习率、权重衰减、训练轮数、随机种子、设备选择和输出目录。`model` 模块负责定义可复用的模型类，至少应清楚区分普通 MLP 基线和利用图结构的图模型。`trainer` 模块负责训练循环、验证、测试、早停、指标记录和模型保存。`utils` 模块负责随机种子、设备选择、目录创建、指标计算、数据划分和通用转换函数。`main.py` 只负责组织流程：解析参数、加载数据、构建模型、启动训练并输出结果。

实现时应避免把数据读取、模型定义、训练循环和结果打印全部堆在一个脚本中。课程大作业需要体现可复现实验流程，因此训练日志、随机种子、数据划分、模型参数和最终指标应可以追踪。任何新增功能都应优先沿用仓库已有数据约定，只有在必要时才重构 `GraphData.py`，并且重构时要保持对 `dataset/` 现有文件格式的兼容。

## 实验与评估约定

节点分类实验应至少报告准确率，并根据实现情况补充宏平均 F1 或加权 F1。若采用半监督设置，应固定训练、验证和测试划分，并在文档或日志中说明划分比例或每类样本数量。模型比较应包含合理基线，例如只使用节点特征的 MLP，以及至少一种使用图结构的模型。报告结论应围绕图结构信息是否提升分类效果、不同数据集上的表现差异以及模型复杂度与性能之间的关系展开。

后续代码检查时，应优先验证以下事项：命令行能指定数据集并完成一次训练；模型输入维度和类别数能从数据中自动推断；CPU 环境可以运行小规模实验；随机种子设置后结果具有基本可复现性；训练、验证和测试指标没有混用；保存文件不会覆盖原始数据。若引入 GPU 支持，应保留 CPU fallback。
